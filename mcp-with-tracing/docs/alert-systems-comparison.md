# 告警系统对比指南

> **目的**: 清晰说明传统告警与智能告警的区别、关系和使用场景  
> **更新日期**: 2026-04-13

---

## 📊 快速对比

| 维度 | 传统告警 (Phase 5) | 智能告警 (Phase 6) |
|------|-------------------|-------------------|
| **核心原理** | 固定阈值规则 | ML 模型自动学习 |
| **配置方式** | 手动设置阈值 | 零配置，自动训练 |
| **检测指标** | 任意自定义指标 | 成功率、延迟、流量、满意度 |
| **适用场景** | SLA 监控、明确业务规则 | 趋势异常、未知模式发现 |
| **误报率** | 较高（需人工调优） | 较低（自适应调整） |
| **响应速度** | 即时（下一个检查周期） | 需要历史数据训练 |
| **检测间隔** | 默认 5 分钟 | 默认 10 分钟 |
| **依赖** | 无额外依赖 | prophet, pyod, pandas, numpy, sklearn |
| **资源占用** | 低 | 中（200-500MB 内存） |
| **启动方式** | AlertMonitorScheduler | SmartAlertManager |

## 🎯 使用场景对比

### 传统告警适合的场景

✅ **明确的业务规则**
```yaml
# 示例: SLA 要求成功率必须 > 99%
alerts:
  - name: "sla-success-rate"
    metric: "success_rate"
    threshold: 0.99
    operator: "lt"
    severity: "critical"
```

✅ **合规性要求**
```yaml
# 示例: P95 延迟不能超过 500ms
alerts:
  - name: "latency-sla"
    metric: "latency_p95_ms"
    threshold: 500
    operator: "gt"
    severity: "warning"
```

✅ **容量规划**
```yaml
# 示例: QPS 超过阈值需要扩容
alerts:
  - name: "high-traffic"
    metric: "request_rate"
    threshold: 100
    operator: "gt"
    severity: "info"
```

✅ **用户满意度监控**
```yaml
# 示例: 平均评分低于 4.0 需要关注
alerts:
  - name: "low-satisfaction"
    metric: "avg_rating"
    threshold: 4.0
    operator: "lt"
    severity: "warning"
```

---

### 智能告警适合的场景

🤖 **未知异常模式发现**
```
场景: 成功率从 98% 突然降到 96%
- 传统告警: 不触发（仍高于 95% 阈值）
- 智能告警: ✅ 触发（检测到显著下降趋势）
```

🤖 **多维度关联异常**
```
场景: 成功率正常，但延迟升高 + 流量下降 + 满意度降低
- 传统告警: 可能都不触发（每个指标都在阈值内）
- 智能告警: ✅ 触发（检测到多维异常组合）
```

🤖 **季节性模式异常**
```
场景: 凌晨 3 点流量突然增加 50%
- 传统告警: 难以设置动态阈值
- 智能告警: ✅ 触发（Prophet 学习到正常夜间模式）
```

🤖 **渐进式退化检测**
```
场景: 延迟每天增加 5%，一周后累计增加 35%
- 传统告警: 可能不触发（未超过绝对阈值）
- 智能告警: ✅ 触发（检测到持续恶化趋势）
```

### 并行运行机制

```python
# 服务器启动时同时初始化两个系统
def main():
    # Phase 5: 传统告警
    monitor = start_alert_monitor(check_interval_minutes=5)
    
    # Phase 6: 智能告警
    smart_manager = SmartAlertManager(detection_interval_minutes=10)
    smart_manager.start_monitoring()
    
    # 两个后台线程独立运行，互不干扰
    mcp.run()
```

## 💡 最佳实践

### 推荐配置策略

#### 1. 同时启用两套系统

```bash
# .env
ALERT_CHECK_INTERVAL_MINUTES=5              # 传统告警：5分钟
SMART_ALERT_CHECK_INTERVAL_MINUTES=10       # 智能告警：10分钟
```

#### 2. 传统告警处理明确规则

```yaml
# config/alerts.yaml
alerts:
  # 关键 SLA
  - name: "critical-success-rate"
    metric: "success_rate"
    threshold: 0.95
    operator: "lt"
    severity: "critical"
    
  # 性能基线
  - name: "high-latency"
    metric: "latency_p95_ms"
    threshold: 500
    operator: "gt"
    severity: "warning"
```

#### 3. 智能告警发现潜在问题

```python
# 无需配置，自动运行
# SmartAlertManager 会自动检测：
# - 成功率异常波动
# - 延迟趋势变化
# - 流量模式异常
# - 满意度下降
```

### 告警去重策略

由于两套系统可能同时检测到同一问题，建议：

1. **告警分组**: 按严重程度分组通知
2. **冷却期**: 相同告警 30 分钟内只发送一次
3. **告警关联**: 识别相关告警，合并通知

```python
# 示例: 告警去重逻辑（需在 Notifier 中实现）
if alert.rule.name in recent_alerts:
    if (now - last_alert_time) < COOLDOWN_PERIOD:
        skip_notification()
```

## 📈 实际案例

### 案例 1: 成功率缓慢下降

**场景**: 成功率从 99% 逐渐降到 97%，用时 3 天

| 时间 | 成功率 | 传统告警 | 智能告警 |
|------|--------|---------|---------|
| Day 1 | 99% | ❌ | ❌ |
| Day 2 | 98% | ❌ | ⚠️ INFO |
| Day 3 | 97% | ❌ | 🚨 WARNING |
| Day 4 | 96% | ❌ | 🚨 CRITICAL |
| Day 5 | 94% | 🚨 CRITICAL | 🚨 CRITICAL |

**结论**: 智能告警提前 2 天发现问题

---

### 案例 2: 突发流量高峰

**场景**: 凌晨 2 点流量突然增加 300%

| 指标 | 传统告警 | 智能告警 |
|------|---------|---------|
| 检测速度 | ✅ 即时（5分钟内） | ⚠️ 需要训练数据 |
| 准确性 | ❌ 可能误报 | ✅ 考虑时间模式 |
| 上下文 | ❌ 仅看绝对值 | ✅ 理解正常模式 |

**结论**: 
- 如果设置了 `request_rate > 100` 规则，传统告警更快
- 如果没有设置规则，智能告警能自动发现异常

---

### 案例 3: 多维度性能退化

**场景**: 
- 成功率: 98% → 96% (正常范围内)
- 延迟: 200ms → 350ms (正常范围内)
- 满意度: 4.5 → 3.8 (轻微下降)

| 系统 | 检测结果 |
|------|---------|
| 传统告警 | ❌ 所有指标都在阈值内，无告警 |
| 智能告警 | 🚨 检测到多维异常组合 |

**结论**: 智能告警发现单一指标正常但整体异常的情况

## 🎓 选择建议

### 只使用传统告警，如果：

- ✅ 你有明确的业务规则和 SLA
- ✅ 你不想安装额外的 ML 依赖
- ✅ 你的系统资源有限
- ✅ 你能接受较高的误报率并手动调优

### 只使用智能告警，如果：

- ✅ 你想零配置自动监控
- ✅ 你关注趋势和模式变化
- ✅ 你有足够的历史数据（24+ 小时）
- ✅ 你可以接受 10 分钟的检测间隔

### 同时使用两套系统（推荐），如果：

- ✅ 你需要全面的监控覆盖
- ✅ 你既要 SLA 监控又要异常发现
- ✅ 你有足够的系统资源
- ✅ 你希望最小化漏报风险

## 🔧 故障排查

### 问题 1: 收到重复告警

**原因**: 两套系统同时检测到同一问题

**解决**:
```python
# 方案 1: 调整检测间隔，错开检查时间
ALERT_CHECK_INTERVAL_MINUTES=5
SMART_ALERT_CHECK_INTERVAL_MINUTES=12  # 避免同时触发

# 方案 2: 在 Notifier 中实现去重逻辑
# 方案 3: 根据告警来源区分处理
if alert.rule.metadata.get('ml_detected'):
    # 智能告警，可能需要不同处理
    pass
```

### 问题 2: 智能告警从未触发

**可能原因**:
1. 历史数据不足（需要至少 50 个数据点）
2. ML 依赖未安装
3. 系统确实没有异常

**检查方法**:
```bash
# 查看启动日志
python -m src.server 2>&1 | grep "Smart ML"

# 检查模型训练状态
# 应该看到类似输出:
# Training anomaly detection models (24h history)...
#   ✓ Trained model for success_rate (144 data points)
```

### 问题 3: 传统告警频繁误报

**解决**:
1. 调整阈值
2. 增加 `window_minutes` 平滑数据
3. 结合智能告警验证（如果智能告警也触发，说明确实有问题）

## ✨ 总结

### 核心要点

1. **两套系统，一个目标**: 全面监控系统健康
2. **互补而非竞争**: 各自擅长不同场景
3. **推荐同时启用**: 最大化覆盖，最小化漏报
4. **共享基础设施**: 通知渠道、告警存储统一管理

### 决策树

```
需要监控什么？
├─ 明确的业务规则/SLA？
│  └─ ✅ 使用传统告警
├─ 未知异常/趋势变化？
│  └─ ✅ 使用智能告警
├─ 都需要？
│  └─ ✅ 同时启用（推荐）
└─ 资源受限？
   ├─ 有 ML 依赖？
   │  └─ ✅ 只用智能告警
   └─ 无 ML 依赖？
      └─ ✅ 只用传统告警
```
